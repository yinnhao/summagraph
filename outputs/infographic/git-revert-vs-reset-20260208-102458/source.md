在 Git 中，git revert 是一个非常重要且安全的“撤销”命令。它的核心作用是：通过创建一个新的提交（Commit），来撤销之前某次提交所做的更改。[1][2]
简单来说，它不是在物理上“删除”历史，而是做一个“反向操作”来抵消掉错误的代码。
1. 核心原理
假设你的提交历史是：A -> B -> C。
如果你发现 B 提交有 Bug，执行 git revert B，Git 会计算出 B 做了哪些改动，然后生成一个新的提交 D，这个 D 的内容正好是 B 的反向。
现在的历史变成了：A -> B -> C -> D。
其中 D 的状态 = C 的状态 - B 的改动。
2. git revert vs git reset
这是最容易混淆的一对命令。它们的区别决定了你在什么时候该用哪一个：
特性	git revert	git reset
操作方式	增加一个新提交，抵消旧提交	移动 HEAD 指针，丢弃之后的提交
历史记录	保留完整的历史，公开透明	抹除历史，看起来像没发生过
安全性	高，不会丢失代码	中/低，如果用了 --hard 且没备份，代码会丢失
适用场景	已推送（Push）到远程的分支	仅限本地尚未推送的分支
金律： 如果代码已经推送到团队共享的远程仓库，永远使用 revert。因为 reset 会改写历史，导致其他队友的本地分支与远程冲突。
